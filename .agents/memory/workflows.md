# Miko Workflow Memory

## Development commands

- Install/run with Bun.
- `bun run dev`: starts client + server in development mode.
- `bun run dev:server`: server only.
- `bun run lint`: Biome check.
- `bun run check`: TypeScript + Vite build.
- `bun test`: test suite.
- `bun run pack:dry`: npm package dry run.

Before opening PRs, normally run at least:

```sh
bun run lint
bun run check
```

Use targeted `bun test path/to/file.test.ts` when changing fragile logic.

## PR conventions

- Use conventional titles: `fix: ...`, `feat: ...`, `refactor: ...`, `chore: ...`.
- Keep PRs focused. Avoid giant rewrite PRs unless explicitly requested.
- Do not push unrelated user changes.
- If the user asks for a PR, create the branch/commit/push/PR and provide the PR URL.
- PR descriptions should use this layout:

```md
## Summary
- One concise bullet explaining the purpose of the PR.

## Changes
- Concrete implementation change.
- Another concrete implementation change.
- Tests or architecture coverage when useful.

## Tests
- `bun run lint`
- `bun run check`
```

Use the actual commands run. If no tests were run, say `Not run (reason)`.

## Release workflow

Miko is published as npm package `miko-code`.

Release shape:

1. Create a release prep PR that bumps `package.json` version.
2. Add professional release notes under `.github/releases/vX.Y.Z.md`.
3. Merge the PR into `main`.
4. Tag the merged main commit from a clean worktree. Do not use `git reset --hard` in release instructions because it can discard unrelated local work.
   ```sh
   git status --short # must be clean before tagging
   git fetch origin main --tags
   git tag -a vX.Y.Z origin/main -m "vX.Y.Z"
   git push origin vX.Y.Z
   ```
5. Tag push triggers `.github/workflows/release.yml`.
6. After every new release, update `.agents/memory/current-state.md` with the latest package version and release chain entry.

Release notes should not be generic GitHub autogenerated text. Be specific about the user-visible behavior, the fixed failure mode, or the concrete improvement. Avoid vague entries like “bug fixes”, “UI polish”, or “stability improvements” unless the note explains exactly what changed. Use concise sections like:

```md
## Fixes

Composer Placeholder Recovery — Clearing all text from the inline composer now restores the placeholder immediately.

## Improvements

Tool Row File Icons — Edit and write tool-call rows now show extension-aware file icons.
```

## npm/GitHub package notes

- Package uses Trusted Publishing through GitHub Actions OIDC.
- npm trusted publisher is configured for repo `Sarp2/miko` and workflow `release.yml`.
- Do not require an npm token in CI unless the publishing model changes.
- The package supports Bun. Users install/run with Bun/npm tooling, but the runtime engine is Bun.

## Local data cleanup

Development data dir: `~/.miko-dev/data`.
Production data dir: `~/.miko/data`.

Be careful when deleting data. The user sometimes explicitly asks to reset both dev and production data. Do not do this unless explicitly requested.
